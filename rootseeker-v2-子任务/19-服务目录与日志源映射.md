# 服务目录与日志源映射

## 1. 目标

本文档用于固化 RootSeeker V2 中 `service_name` 的解析和映射规则。企业级故障排查必须知道一个服务对应哪些日志源、代码仓、索引分区、负责人和环境配置。

强制架构约束：

- 服务目录与日志源映射最终必须通过 `Skill + 内置插件 + MCP` 协同完成。
- 服务目录查询能力必须由内置服务目录插件注册。
- 服务目录查询必须通过 MCP 工具执行，例如 `catalog.resolve_service`。
- Skill 负责声明何时需要解析服务目录，以及解析结果如何参与后续日志、trace、代码检索。
- 默认 Flow 和 Agent Runtime 不得绕过 Skill 和插件直接读取服务目录。

---

## 2. 核心映射

标准映射关系：

```text
tenant + environment + service_name
  -> service catalog entry
  -> log sources
  -> trace source
  -> repositories
  -> zoekt index
  -> qdrant collection
  -> owner team
  -> notification targets
```

标准调用链：

```text
Skill Step
  -> service-catalog bundled plugin
  -> MCP tool: catalog.resolve_service
  -> ServiceCatalogEntry
  -> downstream log/code/trace skills
```

---

## 3. `ServiceCatalogEntry`

必备字段：

- `tenant`
- `environment`
- `service_name`
- `display_name`
- `aliases`
- `owner_team`
- `runtime`
- `language`
- `repositories`
- `log_sources`
- `trace_sources`
- `metric_sources`
- `dependency_group`
- `notification_targets`
- `enabled_skills`
- `enabled_tools`
- `declared_by_plugin`
- `allowed_mcp_tools`
- `catalog_version`
- `audit_policy`

---

## 3.1 内置服务目录插件

必须提供 bundled plugin：

- `plugin_id`: `builtin.service-catalog`
- `kind`: `connector`
- `capabilities`: `catalog.resolve_service`, `catalog.list_log_sources`, `catalog.list_repositories`
- `config_schema`: 服务目录配置 schema

必须注册的 MCP 工具：

- `catalog.resolve_service`
- `catalog.get_log_sources`
- `catalog.get_repositories`
- `catalog.get_notification_targets`

这些工具只能由 Skill Step 调用，不允许被业务模块直接调用。

---

## 4. 服务名解析规则

解析优先级：

1. 告警 payload 显式 `service_name`
2. 告警 rule 绑定服务
3. 日志 tags 中的 service 字段
4. trace service 字段
5. 域名或应用名映射
6. 正则从 message 中提取
7. 人工补录

如果无法解析服务名，Case 必须进入 `need_service_resolution` 状态，不能直接执行代码搜索或日志补全。

---

## 5. 仓库映射

每个服务可对应多个仓库：

- 主应用仓库
- 公共库仓库
- 配置仓库
- 部署仓库

每个仓库需要记录：

- repo url
- default branch
- local path
- index status
- zoekt shard
- qdrant collection
- code owner

---

## 6. 日志源映射

每个服务可对应多个日志源：

- 应用日志
- 错误日志
- 访问日志
- 任务日志
- 中间件日志

每个日志源需要记录：

- provider
- region
- project
- logstore
- query templates
- retention
- data classification
- access policy

---

## 7. 路由与通知映射

服务目录还必须支持：

- 告警路由到团队
- 报告推送到机器人
- 审批请求推送到负责人
- 高危工具调用审批人

---

## 8. 归属任务

- `T1`：补充 `ServiceCatalogEntry` 契约
- `T2`：启动时加载服务目录插件，不直接加载业务服务目录
- `T3`：定义哪些 Skill 可以触发服务目录解析
- `T4`：把服务目录能力注册为 MCP 工具
- `T5`：证据构建时读取服务映射
- `T8`：服务目录能力必须由内置插件声明
- `T9`：通道路由通过 Skill/MCP 使用服务目录
- `T12`：定时刷新索引和服务目录
- `T13`：配置、密钥、权限控制

---

## 9. 验收标准

- `service_name` 能映射到日志源
- `service_name` 能映射到代码仓
- `service_name` 能映射到负责人和通知目标
- 多租户和多环境不会串数据
- 服务目录变更可审计
- 服务目录解析必须能追溯到触发它的 Skill 和 MCP 调用
- 没有内置插件声明的服务目录能力不得被生产链路调用
