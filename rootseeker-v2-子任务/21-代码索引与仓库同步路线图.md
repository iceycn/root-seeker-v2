# 代码索引与仓库同步路线图

## 1. 目标

本文档用于补齐 RootSeeker V2 中代码索引和仓库同步能力，避免只在蓝图中保留 `ZoektAdapter`、`QdrantAdapter` 名称，却没有可执行的索引路线。

强制架构约束：

- 代码索引与仓库同步最终必须通过 `Skill + 内置插件 + MCP` 协同完成。
- 内置索引插件负责声明 RepoSync、Zoekt、Qdrant、LSP 相关能力。
- MCP 负责执行仓库同步、索引状态查询、代码搜索和 LSP 查询。
- Skill 负责声明何时需要同步仓库、何时需要索引、何时需要代码检索，以及结果如何进入 Evidence。
- 默认 Flow、Agent Runtime、RootCauseEngine 不得绕过 Skill 和插件直接调用 Zoekt/Qdrant/LSP。

## 2. 主链路

```text
Skill Step
  -> builtin code-index plugin
  -> MCP repo.sync / index.refresh
  -> RepoSyncCompletedEvent
  -> IndexingQueue
  -> MCP index.zoekt.build
  -> MCP index.qdrant.build
  -> MCP index.status
  -> Skill Step: code evidence search
  -> MCP code.search / lsp.*
  -> CodeEvidence
```

## 3.1 内置代码索引插件

必须提供 bundled plugin：

- `plugin_id`: `builtin.code-index`
- `kind`: `connector`
- `capabilities`: `repo.sync`, `index.refresh`, `code.search`, `lsp.query`

必须注册的 MCP 工具：

- `repo.sync`
- `repo.get_status`
- `index.zoekt.build`
- `index.qdrant.build`
- `index.get_status`
- `code.search`
- `code.read`
- `code.resolve_symbol`
- `lsp.definition`
- `lsp.references`

这些工具必须通过 Skill Step 触发，不能由分析引擎直接调用。

## 4. Zoekt 索引

- 分支策略
- trigram 索引策略

- 失败重试
- 删除索引
- 索引完成事件

## 5. Qdrant 索引

必须定义：

- collection 命名规则
- chunk 策略
- embedding 模型
- 元数据字段
- 删除策略
- 增量更新策略

---

## 6. 与排查链路的关系

代码索引必须服务于：

- `code.search`
- `code.read`
- `code.resolve_symbol`
- `lsp.definition`
- `lsp.references`
- 根因分析中的代码证据

代码证据必须进入 `EvidencePack`，不能只作为工具输出字符串。

代码证据获取链路必须遵循：

```text
Skill Step
  -> MCP code/lsp tool
  -> ToolCallResult
  -> CodeEvidence
  -> EvidencePack
  -> RootCauseEngine
```

## 7. 归属任务

- `T2`：启动骨架中预留索引队列
- `T3`：定义触发代码索引和代码检索的 Skill
- `T4`：MCP 代码工具和索引工具
- `T5`：代码证据进入 Evidence
- `T8`：内置代码索引插件
- `T12`：定时索引刷新
- `T13`：索引状态、日志、审计

---

## 8. 验收标准

- 服务目录能找到仓库
- 仓库能同步到本地
- Zoekt 索引可查询
- Qdrant 向量索引可查询
- 索引失败可重试
- 索引状态可查询
- 代码搜索结果能进入 EvidencePack
- 代码索引和搜索必须能追溯到触发它的 Skill、插件和 MCP 调用
- Zoekt/Qdrant/LSP 不得被 RootCauseEngine 直接调用
