# RootSeeker V2 子任务总览

## 1. 目录定位

本目录用于承接 `rootseeker-v2-重构蓝图.md` 的执行层拆解。

主文档解决的是：

- 为什么重构
- 重构成什么样
- 架构原则是什么

本目录解决的是：

- 先做什么
- 每一步做到什么程度
- 每一步依赖什么
- 每一步产出什么

当前最高优先级：

- 先完成 `root_seek` 现有能力 parity。
- 先钉死现网日志链路数据面。
- 先保证默认故障排查 Flow 可用。
- 暂停继续机械拆分文档，后续只对协议、安全、运行时、日志数据面等复杂模块继续拆。

换句话说，RootSeeker V2 的第一目标不是做一个通用 Agent 平台，而是先完成企业级现网日志链路故障排查系统。

全局调用约束：

- 服务目录、日志数据面、代码索引、通知发送等内置能力都必须通过 `Skill + 内置插件 + MCP` 协同完成。
- 内置 MCP 工具不能被业务模块、默认 Flow、Agent Runtime、RootCauseEngine 直接绕过 Skill 和插件调用。
- Provider、Adapter、SDK 只能作为 MCP 工具背后的实现细节，不能成为上层流程的直接依赖。
- 任意工具调用都必须能追溯到 `Case -> Skill -> Step -> Plugin Capability -> MCP ToolCall`。

## 3. 总体执行顺序

建议严格按以下顺序推进：

1. 先完成 `T1`
2. 再完成 `T2`
3. 尽早完成 `T7`、`T8`、`T9`、`T10` 的架构接口
4. 再完成 `T3` 和 `T4`
5. 再完成 `T5`
6. 并行补齐 `T11`、`T12`、`T13`
7. 补齐 `T14`，统一 Task / Flow Runtime
8. 补齐 `T15`、`T16`、`T17`，把日志数据面、服务目录、代码索引落成可执行任务树
9. 补齐 `T18`，建立回放与评估门禁
10. 最后完成 `T6`

原因：

- `T1` 解决所有后续模块的命名和边界统一问题
- `T2` 解决骨架是否可承载主链路的问题
- `T7` 决定所有排查步骤的执行内核，必须尽早参与骨架设计
- `T8` 决定流程是否能插件化扩展，不应后置
- `T9` 决定异常来源与通知出口，不应只按聊天机器人设计
- `T10` 决定控制面协议和事件推送方式
- `T3` 和 `T4` 分别解决“排查步骤标准化”和“工具接入标准化”
- `T5` 决定系统核心分析质量
- `T11`、`T12`、`T13` 是企业级工程化能力，不是可有可无的附属模块
- `T6` 决定系统能否形成自增长能力
- `T14` 决定所有执行入口是否有统一任务模型
- `T15`、`T16`、`T17` 决定现网日志链路排查的数据面是否真正可落地
- `T18` 决定默认 Flow 和 Skill 质量是否可持续验证

## 7. 二级目录使用方式

每个二级目录下的 `README.md` 现在作为该目录的小目标索引，每个小目标已经拆成独立 `.md` 文件。具体小目标文件继续拆到：

1. 模块任务
2. 输入输出
3. 建议文件
4. 细粒度任务
5. 验收标准
6. 推荐生成顺序

后续如果某个二级模块仍然过大，可以继续在对应目录下按模块新建文件，例如：

- `T3-skill-system/03-registry.md`
- `T3-skill-system/05-composer.md`
- `T4-mcp-plane/04-gateway.md`
- `T5-evidence-analysis/04-context-assembler.md`
- `T5-evidence-analysis/05-root-cause-engine.md`

最终建议形成的生成粒度是：

- 一次只生成一个模块
- 一个模块只改 1 到 3 个文件
- 每个模块都有输入、输出、验收标准

如果某个小目标文件仍然过大，再继续按“一个类/一个函数入口/一个协议对象”拆到三级文件。
