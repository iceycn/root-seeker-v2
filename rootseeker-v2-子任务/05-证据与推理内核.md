# T5 证据与推理内核

## 1. 任务目标

本任务用于定义 RootSeeker V2 的分析核心。

如果说 Skill 系统决定“怎么走流程”，MCP 平面决定“怎么调工具”，那么证据与推理内核决定的就是：

- 证据如何进入系统
- 证据如何被压缩和组织
- 假设如何被提出和验证
- 最终结论如何形成

---

## 2. 范围

本任务覆盖：

- Evidence Center
- Evidence Index
- Context Assembler
- Root Cause Engine
- Reporting

本任务不覆盖：

- 高级模型策略优化
- 全量生产数据接入细节
- 所有专项规则一次性补齐

---

## 3. 输入

- `T1` 的 `EvidencePack`、`ContextWindow`、`CaseReport`
- `T3` 的 Skill 执行轨迹
- `T4` 的 Tool 调用结果

---

## 4. 输出

必须产出：

- 标准证据结构
- Evidence 归档与索引结构
- Context Window 组装规则
- Hypothesis 结构
- RootCauseConclusion 结构
- CaseReport 结构

---

## 5. 一级拆解

### `T5.1` Evidence 标准化

把各类原始数据统一纳入 Evidence 模型。

### `T5.2` Evidence 索引与检索

让证据后续可被搜索和复用。

### `T5.3` Context Window 组装

控制上下文大小与证据顺序。

### `T5.4` Root Cause 推理

完成假设提出、验证和收敛。

### `T5.5` Reporting

形成标准结论输出。

---

## 6. 二级拆解

## 6.1 `T5.1` Evidence 标准化

### 证据类型建议

- `log`
- `trace`
- `metric`
- `code`
- `topology`
- `knowledge`
- `note`

### 类文件任务

- `contracts/evidence.py`
- `evidence/normalizer.py`
- `evidence/evidence_service.py`

## 6.2 `T5.2` Evidence 索引与检索

### 模块任务

- 建立证据主存储
- 建立全文检索字段
- 建立向量检索入口

### 类文件任务

- `storage/evidence_store.py`
- `evidence/indexer.py`
- `evidence/search.py`

## 6.3 `T5.3` Context Window 组装

### 模块任务

- 控制 token 上限
- 控制字符上限
- 对证据做优先级排序
- 对长文本做摘要

### 类文件任务

- `evidence/context_assembler.py`
- `policies/budget.py`
- `analysis/summarizer.py`

## 6.4 `T5.4` Root Cause 推理

### 模块任务

- 生成多个假设
- 评估假设证据支持度
- 去重和收敛
- 生成最终结论

### 类文件任务

- `analysis/root_cause_engine.py`
- `analysis/hypothesis_ranker.py`
- `analysis/verification.py`

## 6.5 `T5.5` Reporting

### 模块任务

- 汇总执行链路
- 汇总结论
- 输出建议动作
- 生成可沉淀摘要

### 类文件任务

- `analysis/reporting.py`
- `contracts/report.py`

---

## 7. 推荐分析流程

建议分析主链路采用以下顺序：

1. 收集原始证据
2. 归一化为 `EvidenceItem`
3. 组装为 `EvidencePack`
4. 生成 `ContextWindow`
5. 输出 `HypothesisSet`
6. 收敛成 `RootCauseConclusion`
7. 输出 `CaseReport`

---

## 8. 风险

- 如果证据类型不统一，后续难以做上下文治理
- 如果 Context Assembler 设计不清晰，后续会重新回到“大上下文硬拼接”
- 如果 Root Cause Engine 不区分“假设”和“结论”，报告可信度会很差

---

## 9. 验收标准

完成本任务时，应该满足：

- 多源证据能被统一收敛
- 上下文窗口可控
- 系统能提出多个假设
- 系统能输出标准结论和建议动作
- 报告能直接作为后续 Skill 沉淀输入

---

## 10. 推荐实施顺序

1. 先做 Evidence 标准化
2. 再做 Context Assembler
3. 再做 Root Cause Engine
4. 最后做 Reporting 和评估
