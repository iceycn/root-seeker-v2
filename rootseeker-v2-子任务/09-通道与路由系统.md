# T9 通道与路由系统

## 1. 任务目标

本任务用于借鉴 OpenClaw 通道与路由设计，为 RootSeeker V2 建立多来源异常消息接入和多出口通知能力。

这里的通道不应局限于企微、飞书、钉钉，也应覆盖：

- 阿里云告警平台
- SLS 告警
- Prometheus / Alertmanager
- Webhook
- OpenAPI
- 企业机器人

参考文档：

- `/Users/beisen/PycharmProjects/openclaw-analysis/03-通道和路由模块分析.md`

---

## 2. 范围

本任务覆盖：

- Channel Registry
- Inbound Adapter
- Outbound Adapter
- Route Resolver
- Session Key
- Binding Routing
- Allowlist / Security

本任务不覆盖：

- 每个企业平台的完整实现
- 复杂 UI 配置

---

## 3. 输入

- 原始告警消息
- Webhook 请求
- API 请求
- 机器人消息
- 通道配置

---

## 4. 输出

- `NormalizedInboundMessage`
- `ResolvedRoute`
- `SessionKey`
- `OutboundTarget`
- `ChannelAuditEvent`

---

## 5. 一级拆解

### `T9.1` 通道契约

定义通道插件的最小接口和可选能力槽。

### `T9.2` Inbound 接入

处理来自告警平台或 Webhook 的异常消息。

### `T9.3` 路由解析

将异常消息路由到对应 Case、Agent、Skill 或团队。

### `T9.4` Outbound 通知

将排查进展和报告推送到机器人或平台。

## 7. 风险

- 如果通道只按聊天工具设计，会无法适配阿里云告警等平台
- 如果 Inbound / Outbound 不分离，通知和告警接入会耦合
- 如果没有 Session Key，重复告警和同一故障聚合会很难处理

---

## 8. 验收标准

- 至少支持一种 Webhook Inbound
- 至少支持一种机器人 Outbound
- 异常消息能被路由到 Case
- 重复告警能被归并到同一 Session 或 Case

---

## 9. 推荐实施顺序

1. 定义 Channel 契约
2. 实现 Inbound Normalizer
3. 实现 Route Resolver
4. 实现 Outbound Target
5. 接入具体平台
