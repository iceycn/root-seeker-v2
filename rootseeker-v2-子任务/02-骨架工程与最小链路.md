# T2 骨架工程与最小链路

## 1. 任务目标

本任务用于搭建 RootSeeker V2 的最小可运行骨架，并验证主架构不是停留在纸面设计。

关键目标不是把所有功能做完，而是跑通一条最小闭环：

`创建 Case -> 选择 Skill -> 调用 Tool -> 生成报告`

---

## 2. 范围

本任务只覆盖：

- 基础目录骨架
- 应用入口
- 容器装配
- 最小 Supervisor
- 最小 Worker
- 最小 API
- 最小 StepExecutor

本任务不覆盖：

- 复杂审批流
- 复杂评估集
- 真实大模型推理
- 全量企业平台接入

---

## 3. 输入

- `T1` 冻结后的对象契约
- 主蓝图中的目录树
- 最小执行链路定义

---

## 4. 输出

必须产出：

- 项目骨架目录
- `apps/api`
- `apps/worker`
- `rootseeker/bootstrap`
- `rootseeker/supervisor`
- `rootseeker/step_engine`
- 一条可运行最小链路

---

## 5. 一级拆解

### `T2.1` 建立工程骨架

搭目录、基础包结构、配置入口。

### `T2.2` 建立容器装配

把各模块连接起来，形成最小运行容器。

### `T2.3` 建立最小 API

对外暴露创建和执行 Case 的接口。

### `T2.4` 建立最小执行链路

从 Case 创建一路贯通到报告生成。

---

## 6. 二级拆解

## 6.1 `T2.1` 工程骨架

### 模块任务

- 创建 `apps/`
- 创建 `rootseeker/`
- 创建 `skills/`
- 创建 `tests/`

### 类文件任务

- `pyproject.toml`
- `README.md`
- `rootseeker/__init__.py`
- `bootstrap/config.py`

## 6.2 `T2.2` 容器装配

### 模块任务

- 建立统一配置读取
- 建立依赖注册
- 建立内建组件初始化顺序

### 类文件任务

- `bootstrap/container.py`
- `bootstrap/config.py`
- `bootstrap/providers.py`

## 6.3 `T2.3` 最小 API

### 模块任务

- 健康检查
- 创建 Case
- 查询 Skill 列表
- 触发 Case 执行

### 类文件任务

- `apps/api/main.py`
- `apps/api/routes/cases.py`
- `apps/api/routes/skills.py`

## 6.4 `T2.4` 最小执行链路

### 模块任务

- CaseSupervisor 接收请求
- SkillComposer 选择一个 Skill
- Planner 生成 Step Graph
- StepExecutor 调用一个内建 Tool
- Reporting 生成报告

### 类文件任务

- `supervisor/case_supervisor.py`
- `case_runtime/planner.py`
- `step_engine/step_executor.py`
- `analysis/reporting.py`

---

## 7. 推荐最小链路说明

建议最小链路只做一条默认技能：

- `log-triage`

该技能只做三件事：

1. 收集基础证据
2. 调用一个默认工具
3. 输出一份简化报告

这样可以验证：

- Skill 选择链路可用
- Tool 调用链路可用
- Report 生成链路可用

---

## 8. 风险

- 如果一开始追求全量骨架，会很快失控
- 如果最小链路没有先跑通，后续所有扩展都会建立在不稳定基础上
- 如果容器装配太复杂，会重复现有 `root_seek` 的问题

---

## 9. 验收标准

完成本任务时，应该满足：

- 项目可以启动
- 可以创建 Case
- 可以挑选一个默认 Skill
- 可以调起一个默认 Tool
- 可以生成一份最小报告

---

## 10. 推荐实施顺序

1. 先搭项目骨架
2. 再搭容器装配
3. 再建 Supervisor
4. 再建最小 API
5. 最后打通最小执行链路
