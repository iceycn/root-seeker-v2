## 2. V1 主链路

V1 的核心链路可以抽象为：

```text
告警/日志输入
  -> IngestEvent
  -> NormalizedErrorEvent
  -> ServiceRouter
  -> LogEnricher
  -> SLS/Trace 查询补全
  -> EvidencePack
  -> Direct Analysis 或 AI Orchestrator
  -> AnalysisReport
  -> Notify
  -> JobQueue / IndexingQueue
```

V2 必须先保证这条链路有等价默认实现，再扩展 Skill、Plugin、Agent Runtime。

迁移约束：

- V1 中的 Provider / Adapter 直连能力，在 V2 中必须收口到 MCP 工具。
- V2 默认链路不能直接调用内置 MCP 工具，必须由 builtin Skill 和 bundled Flow Plugin 触发。
- V2 的每一步 V1 parity 都必须能追溯到 `Case -> Skill -> Step -> Plugin Capability -> MCP ToolCall -> Evidence`。
- 只有底层 MCP 工具实现可以调用 SLS、Zoekt、Qdrant、LSP、通知 SDK 等 Provider。

## 4. V2 必须先达成的 Parity

V2 在进入插件扩展和自动沉淀前，必须先达成以下最小等价能力：

1. 能接收一条阿里云 SLS 告警或标准 Webhook。
2. 能归一化出 `tenant`、`environment`、`service_name`、`severity`、`trace_id`、`timestamp`。
3. 能通过服务目录找到日志源、代码仓库和索引分区。
4. 能基于 `query_key` 和 SQL 模板查询关联日志。
5. 能基于 `trace_id` 和时间窗补全上下文。
6. 能构建 `EvidencePack`。
7. 能走一条不依赖复杂 Agent 的默认直连分析 Flow。
8. 能输出 `CaseReport`。
9. 能推送到至少一个机器人出站通道。
10. 能记录工具调用、查询参数、报告结论和审计事件。

---

## 5. 迁移验收标准

完成 V1 parity 时，必须能用同一条历史故障样本在 V1 和 V2 中跑出可比较结果：

- 输入告警字段一致
- 查询时间窗一致
- SLS 查询模板一致
- trace_id 关联结果数量可解释
- 代码搜索结果可解释
- 报告结论和建议动作可人工对比
- 审计记录完整

---

## 6. 风险

- 如果不做 parity，V2 很可能变成“架构更漂亮但排查能力退化”的系统。
- 如果默认 Flow 不先落地，Skill/Plugin 会成为使用门槛。
- 如果 SLS/trace/服务映射没有契约化，企业现网接入会长期依赖人工配置和隐式约定。
