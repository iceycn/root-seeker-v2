# 日志数据面契约

## 1. 文档目标

本文档固化 RootSeeker V2 **现网日志数据面**的协议边界：日志从哪查、怎么查、查完变成什么结构、如何审计与回放。

它不定义具体云厂商 SDK 细节，但要求所有实现满足同一套**对外契约**与**调用链约束**。

---

## 2. 强制架构约束

与全局约束一致（见 `00-子任务总览.md`、`20-内置默认故障排查Flow.md`）：

- 任意生产路径上的日志查询必须经过：**Skill Step → 内置日志插件（bundled）→ MCP 工具 → Provider/Adapter（实现细节）**。
- **禁止**默认 Flow、Agent Runtime、RootCauseEngine、业务模块**直接**调用 `SlsAdapter`、其它日志 SDK 或未注册的工具。
- 所有对外可见的日志能力必须通过已注册的 MCP 工具暴露，例如：
  - `log.query_by_trace_id`
  - `log.query_by_template`
- 每一次查询必须可追溯到：`case_id`（或等价上下文）、`skill_name`、`plugin_id`、`mcp_tool_name`，并产生可持久化的 **query_key**（见第 7 节）。

---

## 3. 范围与非目标

### 3.1 范围

- 日志源（LogSource）与 `ServiceCatalogEntry.log_sources` 的衔接。
- 查询请求、时间窗、模板与参数、trace 维度查询。
- 统一结果模型 `LogQueryResult` / `LogRecord`（与代码中 `rootseeker/contracts/log_query.py` 对齐）。
- 审计字段、query_key 规则、截断与脱敏原则。
- 与 `EvidenceItem`（`EvidenceType.LOG`）的映射关系。

### 3.2 非目标

- 具体 SLS Project/Logstore 的运维与配额治理细则。
- 全文检索引擎选型与索引构建（归「代码索引」任务树）。
- 自然语言生成 SQL（若引入，须独立策略模块，不得绕过 MCP）。

---

## 4. 核心对象与命名


| 名称                          | 说明                                                            |
| --------------------------- | ------------------------------------------------------------- |
| `LogSource`                 | 描述「到哪查」：类型（如 sls）、endpoint、project、store、topic、鉴权引用等（见第 5 节）。 |
| `LogQueryTemplate`          | 预置或注册的查询模板：`template_id`、参数 schema、渲染后为可执行查询语句或等价 API 调用描述。   |
| `LogQueryByTraceIdRequest`  | 按分布式 trace_id 拉取相关日志的请求契约。                                    |
| `LogQueryByTemplateRequest` | 按模板 ID + 参数发起查询的请求契约。                                         |
| `LogRecord`                 | 单行归一化日志（时间、级别、message、trace_id、raw 扩展）。                       |
| `LogQueryResult`            | 一次查询的完整结果：`query_key`、`records`、`truncated`、元数据。              |


代码落点（当前仓库）：

- `rootseeker/contracts/log_query.py`：`LogQueryByTraceIdRequest`、`LogQueryByTemplateRequest`、`LogRecord`、`LogQueryResult`。

后续可拆文件（见 `24-T15-日志数据面任务树.md`）：

- `contracts/log_source.py`：`LogSource`
- `contracts/log_query.py`：模板相关扩展（如 `LogQueryTemplate` 定义）

## 5. LogSource 与目录映射

### 5.1 来源

`LogSource` 由 **服务目录**解析得到：路径为

`tenant + environment + service_name → ServiceCatalogEntry → log_sources[]`。

单条 `LogSource` 建议至少包含：


| 字段                            | 必填     | 说明                                               |
| ----------------------------- | ------ | ------------------------------------------------ |
| `type`                        | 是      | 如 `sls`、`elasticsearch`、`loki`；实现可扩展，须在插件侧注册解析器。 |
| `source_id`                   | 是      | 稳定 ID，用于审计与回放对齐。                                 |
| `display_name`                | 否      | 展示用。                                             |
| `endpoint` / `region`         | 视 type | 访问维度。                                            |
| `project` / `store` / `topic` | 视 type | 日志库定位。                                           |
| `default_query_language`      | 否      | 如 `sql`、`lucene`；默认与模板一致即可。                      |
| `secret_ref`                  | 视部署    | 密钥引用，不内联明文。                                      |
| `tags`                        | 否      | 环境、集群、泳道等。                                       |


### 5.2 解析责任

- **谁解析**：内置 `builtin.log-query` 插件 + MCP 实现内部逻辑；**不允许** Flow 业务代码手写 project/store。

---

## 6. 时间窗（Time Window）

### 6.1 原则

- 所有日志查询必须绑定**显式或可推导**的时间范围 `[time_from, time_to]`（UTC 或带时区 `datetime`，序列化时建议 ISO8601）。
- 若请求未传时间窗，由 **time_window 策略模块**（规划在 `rootseeker/log_data/time_window.py`）根据告警时间、Case `metadata`、租户策略计算默认值，并写入 `LogQueryResult.metadata["resolved_time_window"]` 供审计与回放。

### 6.2 建议元数据

- `metadata["alarm_time"]`：告警锚点时间。
- `metadata["lookback_minutes"]`：回溯分钟数（策略产出）。
- `metadata["tenant_policy_ref"]`：策略版本引用（可选）。

---

## 7. query_key 与审计

### 7.1 query_key 定义

`LogQueryResult.query_key`：**稳定、可复算、短**的字符串，用于：

- 审计日志关联（同一逻辑查询多次执行应得到相同 query_key，若参数与时间窗一致）。
- 回放用例对齐（期望结果与 query_key 绑定）。

### 7.2 推荐构成

对 `log.query_by_trace_id`：

- 建议格式：`trace:{trace_id}@{source_id}` 或对 `(trace_id, source_id, normalized_time_window)` 的规范化哈希（取前 16～32 位十六进制）。

对 `log.query_by_template`：

- 建议格式：`tpl:{template_id}:{hash(parameters_norm, time_window_norm, source_id)}`。

**禁止**将完整 SQL/原始鉴权信息写入 query_key；参数应做规范化（键排序、空值剔除）后再哈希。

### 7.3 审计必录字段

每次 MCP 日志工具调用，审计 `detail` 建议至少包含：

- `case_id`
- `skill_name`
- `plugin_id`
- `mcp_tool_name`
- `query_key`
- `log_source_id` 或等价定位
- `record_count`、`truncated`
- `duration_ms`（可选）

---

## 8. SQL / 模板与参数

### 8.1 LogQueryTemplate（逻辑契约）


| 字段                 | 说明                                                             |
| ------------------ | -------------------------------------------------------------- |
| `template_id`      | 全局唯一，如 `default.error_rate_by_service`。                        |
| `version`          | 模板版本，变更后回放集需同步。                                                |
| `parameter_schema` | JSON Schema 或等价描述，用于校验 `LogQueryByTemplateRequest.parameters`。 |
| `render_kind`      | 如 `sls_sql`、`lucene`；由 `log_data/query_renderer.py` 渲染。        |
| `template_body`    | 模板正文（占位符由渲染器替换）。                                               |


### 8.2 参数校验

- 渲染前必须通过 schema 校验；失败返回**可审计**错误（`ToolCallResult` / `ToolError`），不得静默吞掉。

---

## 9. trace_id 维度

### 9.1 查询语义

- `log.query_by_trace_id`：在已解析的 `LogSource` 上，按 `trace_id` 检索关联日志行；可叠加 `service_name`、时间窗、limit。
- `trace_id` 来源优先级（与 `19-服务目录与日志源映射.md` 一致）：告警显式字段 → 日志 tags → trace 侧回填 → 规则提取。

### 9.2 与 Trace 数据面边界

- **分布式调用链**的拓扑由 `trace.get_chain`（Trace 面 MCP）负责；日志面只保证「同一 trace_id 的日志证据」可被拉取并汇入 `EvidencePack`。

---

## 10. 结果模型与截断

### 10.1 LogRecord

- `timestamp`：归一化时间。
- `message`：主文本。
- `level`：级别（若源无可为空）。
- `trace_id`：若该行可关联 trace。
- `raw`：未归一化的原始字段子集（注意脱敏，见第 11 节）。

### 10.2 LogQueryResult

- `records`：本次返回行列表。
- `truncated`：是否因 limit / 配额截断；为 `true` 时必须在 `metadata` 中说明原因与上限。
- `metadata`：可含 `resolved_time_window`、`sls_request_id`（非密钥）、后处理统计等。

---

## 11. 后处理、脱敏与合规

- **post_filter**（规划在 `rootseeker/log_data/post_filter.py`）：按租户策略剔除字段、打码手机号/身份证/Token 等。
- 进入 `EvidenceItem.content` 的数据应为**已脱敏或可公开**视图；原始 `raw` 默认不进入 RootCause 上下文，除非策略明确允许。
- 脱敏规则版本号建议写入 `EvidencePack.metadata` 或审计 detail。

---

## 12. 映射到 Evidence

- 每条 `LogQueryResult` 经 `log_data/evidence_mapper.py` 映射为一条或多条 `EvidenceItem`：
  - `type = EvidenceType.LOG`
  - `source = mcp_tool_name`（如 `log.query_by_trace_id`）
  - `content` 建议包含：`query_key`、`log_source_id`、`record_count`、`truncated`、摘要片段（非全量 raw）

保证：**任意查询结果可还原为证据且可审计**（见 `24-T15-日志数据面任务树.md` 验收标准）。

---

## 13. 回放与评估

- 回放样本（见 `22-回放与评估基准.md`）应包含：`alert_payload`、`case_request`、可选 `log_samples`；执行时应对齐相同 `query_key` 下的行为（mock 或录制）。
- 评估维度至少覆盖：是否查询必要日志、是否截断可感知、是否审计完整、是否越权（见回放文档「安全性」）。

---

## 14. 标准调用链（摘要）

```text
Skill Step
  -> builtin.log-query 插件
  -> MCP log.query_by_trace_id | log.query_by_template
  -> Log Provider Adapter（实现细节）
  -> LogQueryResult
  -> EvidenceItem（LOG）
```

---

## 15. 验收清单（本契约）

- 所有生产日志查询均带 `skill_name`、`plugin_id`、`mcp_tool_name` 审计维度。
- 每次查询生成稳定 `query_key`，且不含密钥与完整 SQL 明文。
- 时间窗可解析、可记录、可回放。
- `LogQueryResult` 可映射为 `EvidenceItem`（LOG）。
- 默认 Flow 不直接依赖 SLS SDK；仅 MCP 背后实现可依赖。
